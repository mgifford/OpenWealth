Anchor Links Accessibility Best Practices

A reference guide demonstrating how to add heading anchor links that are keyboard accessible, screen-reader friendly, and meet WCAG 2.2 AA requirements.

What are anchor links? §

Anchor links (also called heading permalinks or deep links) are links that point directly to a specific section on a page using a fragment identifier — the part of a URL after the # symbol.

For example, the URL https://example.com/page.html#what-are-anchor-links would scroll the browser to the element with id="what-are-anchor-links".

Anchor links are useful for:

  • Sharing a direct link to a specific section of a long page
  • Building accessible table-of-contents navigation
  • Letting users bookmark or return to a specific part of documentation
  • Providing "skip to section" targets in long-form content

Why accessibility matters §

Anchor links can create accessibility barriers if not implemented carefully:

  • Screen reader users need a clear, descriptive link name — not just an ambiguous # symbol that is read aloud in isolation.
  • Keyboard-only users must be able to reach and activate the anchor link without a mouse.
  • Low-vision users need sufficient colour contrast on the link text and a visible focus indicator.
  • Cognitive accessibility — the visual indicator (§ or #) should appear consistently and predictably so users learn to expect it.

Implementation pattern §

The recommended HTML pattern places the anchor link inside the heading element, immediately after the heading text. Using aria-label on the link provides a meaningful accessible name to screen readers, independent of the visible § character.

HTML § 

<!-- ✅ Recommended pattern -->
<section aria-labelledby="section-id">
  <h2 id="section-id">
    Section Title
    <a class="heading-anchor"
       href="#section-id"
       aria-label="Link to Section Title">§</a>
  </h2>
  <p>Section content...</p>
</section>

Key points:

  • The id is placed on the heading element, not the section.
  • The containing <section> uses aria-labelledby pointing to the heading id.
  • The <a> element has an aria-label that clearly describes its purpose.
  • The visible symbol § (section sign) conveys meaning without relying on colour.
  • The link href points to the same #id on the heading.

Choosing aria-label values §

There are two common conventions for aria-label on heading anchor links. Choose the one that fits your maintenance model:

  • Descriptive (recommended for most cases): "Link to [Section Name]" — unambiguous when screen readers list all links on a page (e.g., using NVDA's link list or VoiceOver's rotor). The trade-off is that the label must be updated whenever the heading text changes.
  • Generic (lower maintenance): "Permalink" or "Link to this section" — never needs updating, and still works because the screen reader announces the surrounding heading as context. The trade-off is that links listed out of context (outside the heading) are less descriptive. 
<!-- ✅ Descriptive: clear when listed out of context -->
<a href="#results" aria-label="Link to Results section">§</a>

<!-- ✅ Generic: lower maintenance, relies on heading context -->
<a href="#results" aria-label="Permalink">§</a>

<!-- ❌ Avoid: no accessible name at all -->
<a href="#results">§</a>

<!-- ❌ Avoid: aria-hidden removes the link from the accessibility tree -->
<a href="#results" aria-hidden="true">§</a>

CSS pattern §

The anchor link is hidden by default (opacity 0) and revealed when the user hovers over the heading, when focus moves into the heading, or when the link itself receives focus. This is the most common and accessible pattern. 

.heading-anchor {
  display: inline-block;
  margin-left: 0.4em;
  color: var(--muted);
  text-decoration: none;
  font-size: 0.85em;

  /* Hidden by default, revealed on hover/focus */
  opacity: 0;
  transition: opacity 160ms ease;
  vertical-align: middle;
}

/* Show when heading is hovered or focus is inside heading */
h2:hover .heading-anchor,
h2:focus-within .heading-anchor,
.heading-anchor:focus {
  opacity: 1;
}

/* Accent colour and underline when active */
.heading-anchor:hover,
.heading-anchor:focus {
  color: var(--accent);
  text-decoration: underline;
  opacity: 1;
}

/* Respect user preference for reduced motion */
@media (prefers-reduced-motion: reduce) {
  .heading-anchor {
    transition: none !important;
  }
}

Important: Using opacity: 0 (rather than display: none or visibility: hidden) keeps the element in the accessibility tree and tab order at all times, so keyboard-only users can still focus and activate the link. The :focus selector in the CSS then makes the link visually visible whenever it receives keyboard focus — ensuring keyboard users always see the link while navigating by Tab, even without hovering.

Do's and don'ts §

✅ Do

  • Use aria-label on anchor links to describe their purpose
  • Place the anchor link inside the heading element
  • Use opacity: 0 to hide visually while keeping in the tab order
  • Reveal the anchor link on heading hover and on focus
  • Use unique, descriptive id values on headings
  • Add aria-labelledby to the containing <section>
  • Respect prefers-reduced-motion by removing the transition
  • Ensure sufficient colour contrast on the link text (WCAG 1.4.3)

❌ Don't

  • Use aria-hidden="true" on anchor links (hides from screen readers)
  • Use display: none or visibility: hidden (removes from tab order)
  • Rely solely on colour to convey the link is clickable
  • Use duplicate id values on the same page
  • Place the anchor link outside the heading (breaks the relationship)
  • Use generic labels like "link" or "#" as the accessible name
  • Forget to test with keyboard-only navigation

WCAG criteria reference §

The following WCAG 2.2 success criteria apply to heading anchor links:

  • 1.4.3 Contrast (Minimum) — Level AA: The visible anchor link symbol must have a colour contrast ratio of at least 4.5:1 against its background.
  • 2.1.1 Keyboard — Level A: All functionality (including activating the anchor link) must be operable via keyboard.
  • 2.4.3 Focus Order — Level A: The link must receive focus in a logical order consistent with the reading order.
  • 2.4.4 Link Purpose (In Context) — Level AA: The purpose of each anchor link must be determinable from the link text alone or from its context (use aria-label for this).
  • 2.4.7 Focus Visible — Level AA: The anchor link must have a visible focus indicator when keyboard-focused.
  • 4.1.2 Name, Role, Value — Level A: The link element must have an accessible name (provided via aria-label).

Live examples §

The headings on this page itself use the patterns described above. Try the following to verify accessibility:

  1. Hover test: Hover over any section heading above — a § symbol should appear to the right of the heading text.
  2. Keyboard test: Press Tab to move focus through the page. When focus lands on a heading anchor link, the § symbol should become visible and the link should have a visible focus ring.
  3. URL test: Click any § anchor link — the page URL should update to include the section's fragment (e.g., #live-examples). Sharing this URL should scroll a new visitor directly to that section.
  4. Screen reader test: Navigate by links (e.g., using NVDA's Insert+F7 or VoiceOver's VO+U). Each anchor link should be announced as "Link to [Section Name]" — not just "§" or "section sign".
